# AI SDK Streaming Format Mismatch Troubleshooting

## Issue Description
**Date:** 2026-01-17  
**Component:** AI Chat Module (Frontend & Backend)  
**Error:** `Type validation failed` in frontend console. `invalid_union` expecting `response.output_text.delta` but invalid input received.

### Symptoms
1. Frontend `useChat` hook received errors indicating a type mismatch.
2. AI response was not displaying in the UI, although `onFinish` callbacks seemed to trigger in some cases.
3. Logs indicated the backend was sometimes returning `text_completion` format (Legacy) or `[Stream Data]` was not being parsed correctly by the frontend.

## Root Cause Analysis
1. **API Protocol Mismatch:** 
   - The AI SDK v6's `openai(model)` function defaults to using the new **OpenAI Responses API** (`/v1/responses`).
   - Our local LLM backend (running on port 8045) supports `/v1/responses`, but there were compatibility issues when interacting with the AI SDK's parsing logic (potentially due to strict format validation or fallback behaviors).
   - The frontend `useChat` hook (from pure `@ai-sdk/react`) expects a specific stream format generated by `toUIMessageStreamResponse()`.

2. **Backend Compatibility:**
   - While the local backend *technically* returned correct SSE events for `stream: true` on `/v1/responses`, the integration was unstable or misconfigured for the AI SDK's default Responses API client.
   - The Chat Completions API (`/v1/chat/completions`) is the older, more widely supported standard for OpenAI-compatible proxies.

## Solution
We explicitly configured the AI SDK to use the **Chat Completions API** instead of the default Responses API.

### Code Changes
**File:** `frontend/src/app/api/chat/route.ts`

**Before (Default - Responses API):**
```typescript
const result = streamText({
    model: openai(MODEL_NAME), // Uses /v1/responses
    // ...
});
```

**After (Fix - Chat Completions API):**
```typescript
const result = streamText({
    model: openai.chat(MODEL_NAME), // Uses /v1/chat/completions
    // ...
});
```

## Why This Works
By using `.chat()`, we force the SDK to use the `/v1/chat/completions` endpoint. This endpoint generally has better stability and compatibility with third-party or local OpenAI-compatible LLM servers. The `toUIMessageStreamResponse()` function is still used to format the outgoing stream to the frontend, but the *source* data from the LLM is now fetched via the robust Chat API protocol.

## Recommendations for Future
- If connecting to non-OpenAI official endpoints (e.g., local LLMs, open-source proxies), prefer using `openai.chat(model)` to ensure maximum compatibility.
- Use `openai(model)` (Responses API) primarily when connecting directly to OpenAI's official API or when specific Responses API features are required and confirmed supported by the backend.
